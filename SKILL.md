# 产品思考三步法

使用此skill将模糊的产品问题转化为清晰可行的产品方案。

此skill结合了三种核心思维模式，按实用顺序执行：

1. **苏格拉底式提问** - 拆解模糊问题，明确真实需求
2. **第一性原理** - 挖掘底层逻辑，回归事实本质
3. **奥卡姆剃刀** - 确定最小可行方案，去除冗余复杂度

## 核心目标

产生能够：

1. 识别真实的用户问题和需求
2. 暴露隐藏的假设和惯性思维
3. 将问题还原为用户行为、事实和约束
4. 比较方案的复杂度和假设负担
5. 去除虚假复杂度
6. 输出清晰的产品方案、MVP定义或下一步行动

优秀的回答应该让产品经理感觉：

- "真实问题现在清楚了"
- "我之前忽略了这个关键假设"
- "这比我想象的更简单"
- "我知道下一步该做什么了"

## 操作姿态

默认先澄清，再自信。

始终问：

- 用户真正想要解决的问题是什么？
- 哪些部分是模糊的、继承的、未经验证的？
- 哪些事实和硬约束真正重要？
- 哪个方案需要更少的不必要假设？
- 什么是仍然尊重现实的最简路径？

## 最佳使用场景

此skill特别适用于：

- 新产品构思和验证
- 功能优先级决策
- 产品需求分析（PRD撰写前）
- MVP范围界定
- 用户反馈分析和需求提取
- 产品路线图规划
- 竞品分析和差异化定位
- 产品复杂度简化
- 增长策略设计

## 不适合的场景

不要将此skill用于：

- 简单的事实查询
- 纯粹的执行任务
- 明显的低风险选择
- 用户明确只需要常规指导而不需要重新框架的情况

## 工作流程路由

路由到最小有用的工作流程。

- 问题模糊、超载、框架不当 → `Workflows/Clarify.md`
- 需要拆解为事实、约束和机制 → `Workflows/Deconstruct.md`
- 有太多解释、步骤、功能或移动部件 → `Workflows/Simplify.md`
- 需要产品方案、MVP定义或决策 → `Workflows/Decide.md`

对于非平凡问题，使用完整序列：

1. 澄清（Clarify）
2. 拆解（Deconstruct）
3. 简化（Simplify）
4. 决策（Decide）

对于较简单的问题，压缩序列但保留逻辑。

## 模式选择

选择仍然能改进决策的最轻响应模式。

### 模式A：快速重构

用于简短问题，例如：

- "我遗漏了什么？"
- "我是不是想得太复杂了？"
- "这个功能真的必要吗？"
- "哪个方案更合理？"

输出：

- 真实问题
- 隐藏假设
- 主要约束
- 更简单的结论
- 下一步行动

### 模式B：结构化产品分析

用于新产品、功能设计、MVP界定、需求分析。

输出：

- 真实用户目标
- 关键假设
- 基本事实
- 硬约束
- 软约束
- 简化的方案选项
- 产品方案/MVP建议
- 第一个验证步骤或下一步行动

### 模式C：需求审计

当产品经理提出的产品需求、功能设计或决策路径可能薄弱、混乱或过度构建时使用。

输出：

- 真实问题
- 薄弱假设
- 缺失证据或矛盾
- 更简单的解释或设计
- 优化后的判断

不要在较短的分析足够时强制长篇分析。

## 必要的推理规则

### 1. 解决前先澄清

不要很好地解决错误的问题。

用结果术语重述问题，而不仅仅是用户当前的框架。

示例：

- "我应该开发这个功能吗？"可能真正意味着"实现用户结果的最好交付机制是什么？"
- "如何让产品更专业？"可能真正意味着"如何增加信任、清晰度、转化率或权威性？"
- "为什么这个功能很难用？"可能真正意味着"哪部分实际上是瓶颈？"

如果问题在错误层面框架，请说明并纠正。

### 2. 显式暴露假设

在以下方面寻找假设：

- 用户的措辞
- 继承的工作流程
- 行业规范
- 默认最佳实践
- 偏好的工具或格式
- 表现为逻辑的情感偏好

有用的自我提示：

- 什么被视为必要？
- 这个结论要成立必须满足什么条件？
- 什么在没有证据的情况下被接受？
- 哪个假设承载最大权重？

### 3. 将问题还原为事实和约束

将问题分解为：

- 用户目标
- 用户行为
- 激励机制
- 成本结构
- 时间要求
- 依赖关系
- 信息流
- 因果驱动因素
- 法律或技术边界
- 真实风险

偏好机制而非叙述。

不要说"市场就是这样运作的"，除非你解释实际机制。

### 4. 区分硬约束和软约束

#### 硬约束
除非有证据表明否则，否则将这些视为真实：

- 物理限制
- 法律或法规限制
- 硬技术边界
- 固定预算或能力
- 现实中已固定的不可移动截止日期

#### 软约束
默认将这些视为可挑战：

- 行业惯例
- 遗留流程步骤
- 默认工具选择
- 当前组织边界
- 美学期望
- 现有架构
- "通常都是这样做的"

永远不要在没有正当理由的情况下将软约束呈现为不可变。

### 5. 偏好更低假设负担

比较解释或选项时，偏好：

- 需要更少推测假设的
- 引入更少移动部件的
- 添加更少协调成本的
- 用更少结构保持真实目标的
- 仍然充分符合事实和约束的

不要通过删除现实来简化。简单性必须保持充分性。

### 6. 以决策结束

始终以以下之一结束：

- 产品方案建议
- MVP定义
- 功能优先级列表
- 决策规则
- 最小有用实验
- 第一个实施步骤
- 下一步需要回答的简短问题列表

不要仅以抽象反思结束。

## 如果信息不完整

不要停滞。相反：

1. 命名关键模糊性
2. 陈述最可能的解释
3. 明确做出最小必要假设
4. 用最佳当前解释继续
5. 说明哪个事实最能改变建议

## 质量标准

使用此skill产生的优秀回答应该：

- 识别真实的决策或问题
- 暴露隐藏假设
- 将事实与惯性分开
- 减少不必要的复杂性
- 保持充分性
- 留给用户更清晰的下一步

如果回答听起来聪明但不能改进用户的产品决策质量，那还不够好。
