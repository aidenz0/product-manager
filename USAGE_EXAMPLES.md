# 使用示例

## 示例1：新功能评估

**用户提问：**
```
我们的用户一直要求添加社交分享功能。我们应该做吗？
```

**使用产品思考三步法：**

### 第一步：澄清（Clarify）
- **表面问题**：是否应该添加社交分享功能？
- **模糊术语**："用户要求" - 多少用户？什么类型？如何要求的？
- **隐藏假设**：
  - 社交分享会带来增长
  - 用户要求的功能就是他们需要的
  - 竞争对手有所以我们也要有
- **真实目标**：用户想要与他人分享内容，但需要理解为什么、分享什么、与谁分享

### 第二步：拆解（Deconstruct）
- **基本事实**：
  - 15个用户在过去3个月提到分享
  - 当前没有分享功能
  - 用户流失率在30天达到60%
- **硬约束**：开发团队已满负荷2个月
- **软约束**："社交产品应该有分享"的行业规范
- **因果结构**：高流失率 → 用户找不到价值 → 可能缺乏社交认可或分享机制

### 第三步：简化（Simplify）
- **假设负担**：
  - 完整社交分享系统：高（假设用户会分享、假设分享带来增长）
  - 简单分享链接：低（测试用户是否真的想分享）
- **可移除复杂度**：多平台集成、分享预览定制、分享分析

### 第四步：决策（Decide）
- **建议**：不立即构建完整社交分享系统
- **最小实验**：
  1. 在3-5个关键页面添加简单的"复制链接"按钮
  2. 追踪点击率和分享后用户行为
  3. 访谈5-10个用户，询问分享动机
- **成功指标**：
  - 10%用户点击分享按钮
  - 分享用户留存率提高20%
- **下一步**：本周实施复制链接按钮，2周后评估数据

## 示例2：MVP范围界定

**用户提问：**
```
我们正在构建一个远程团队协作工具。MVP应该包含什么功能？
```

**使用产品思考三步法：**

### 第一步：澄清（Clarify）
- **表面问题**：MVP功能列表是什么？
- **真实目标**：用最少功能验证什么核心价值主张？
- **重构后问题**：远程团队协作的最大痛点是什么，我们如何用最简方案验证我们能解决它？

### 第二步：拆解（Deconstruct）
- **真实目标**：帮助远程团队更高效协作
- **问题组件**：
  - 用户：分布式团队、跨时区
  - 痛点：时差、沟通异步、信息分散
- **基本事实**：
  - 市场已拥挤（Slack、Teams、Notion）
  - 初始目标：小型技术团队（5-20人）
- **硬约束**：
  - 2个人开发团队
  - 3个月开发时间
  - 自力更生（无外部资金）

### 第三步：简化（Simplify）
- **假设负担**：
  - 全功能协作平台：高（假设可以与成熟产品竞争）
  - 专注单一用例：低（假设在特定场景可以更好）
- **可移除复杂度**：
  - 移动App（Web优先）
  - 视频会议（集成现有工具）
  - 复杂权限系统

### 第四步：决策（Decide）
- **MVP建议**：专注"异步日常更新"单一用例
- **MVP功能**：
  - 每日站队（文字版）
  - 时区感知的自动提醒
  - 简单进度更新（红/黄/绿状态）
  - 基础历史记录
- **排除的功能**：实时聊天、文件共享、视频会议、移动App
- **验证计划**：
  - 招募10个远程团队试用beta版
  - 目标：40% DAU、60%留存、20%愿意付费≥$10/月
- **下一步**：创建原型，开始用户访谈验证"异步日常更新"是否真的是痛点

## 示例3：产品复杂度简化

**用户提问：**
```
我们的产品变得很复杂，用户抱怨难用。我们该怎么办？
```

**使用产品思考三步法：**

### 第一步：澄清（Clarify）
- **表面问题**：如何简化复杂产品？
- **模糊术语**："复杂"、"难用" - 具体指什么？
- **真实目标**：提高用户可用性和满意度，可能通过减少而非优化

### 第二步：拆解（Deconstruct）
- **基本事实**：
  - 产品有47个功能
  - 新用户激活率：15%完成首次关键任务
  - 用户反馈："太多选项"、"不知道从哪开始"
- **硬约束**：有些用户依赖某些功能
- **因果结构**：功能太多 → 选择过载 → 用户困惑 → 放弃
- **虚假复杂度**：复杂度是认知性的（用户心理负担），而非技术性

### 第三步：简化（Simplify）
- **可移除**：
  - 使用率<5%的功能（12个功能）
  - 重复功能（3组）
  - "以防万一"功能（8个）
- **必须保留**：核心用户流程（前20%功能覆盖80%使用）

### 第四步：决策（Decide）
- **建议**：渐进式简化，而非大重构
- **三阶段计划**：
  - **阶段1（立即）**：隐藏低使用率功能，简化新手引导
  - **阶段2（1个月内）**：正式移除使用率<2%的功能，合并重复功能
  - **阶段3（3个月内）**：基于用户行为重新设计核心流程
- **验证**：测量激活率、支持票据、用户满意度
  - 目标：激活率提升至35%，支持请求减少30%
- **下一步**：本周实施阶段1，2周后评估数据

## 关键要点

所有示例共享的模式：

1. **从表面问题到真实问题** - 重新框架用户的初始问题
2. **暴露假设** - 明确说出未声明的假设
3. **基于事实** - 使用数据、观察、证据，而非观点
4. **简化思维** - 识别和移除不必要的复杂度
5. **可操作输出** - 以具体的下一步、实验、验证计划结束

**记住：目标是更好的产品决策，而不是更聪明的分析。**
