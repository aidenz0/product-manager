# 测试示例：产品思考三步法

## 测试场景

假设你是一名产品经理，遇到以下问题：

**问题：**
"我们的竞品刚刚推出了AI推荐功能，我们应该也做这个吗？"

## 预期输出结构

当使用product-thinking skill时，应该按照以下结构思考：

### 1. 澄清（Clarify）
- 表面问题：是否应该做AI推荐功能？
- 模糊术语："AI推荐" - 具体指什么？什么样的推荐？
- 隐藏假设：
  - 竞品有的我们就必须有
  - AI推荐能解决用户问题
  - 我们有能力和资源做AI
- 真实目标：用户需要更好的内容发现体验，但AI可能不是唯一或最佳方案

### 2. 拆解（Deconstruct）
- 基本事实：
  - 竞品有AI推荐
  - 我们不知道用户是否使用或喜欢竞品的AI推荐
  - 我们不知道用户当前如何发现内容
  - AI需要数据、算法、技术能力
- 硬约束：
  - 我们没有AI团队
  - 没有历史用户行为数据训练模型
  - 有限的开发资源
- 软约束：
  - "AI是趋势"的行业 hype
  - "需要保持竞争力"的恐惧
- 因果结构：
  - 用户可能找不到想要的内容 → 流失
  - 但也可能：内容分类/标签/搜索更好就能解决

### 3. 简化（Simplify）
- 假设负担：
  - 完整AI推荐系统：极高（假设有数据、算法、用户会信任、会有效）
  - 改进内容分类/搜索：低（直接解决"找不到内容"问题）
- 可移除复杂度：
  - 机器学习算法
  - 个性化模型
  - 实时推荐引擎
- 最简方案：
  - 改进内容标签和分类
  - 添加"热门内容"或"编辑推荐"
  - 改进搜索功能

### 4. 决策（Decide）
- 建议：**不要**立即构建AI推荐系统
- 理由：
  - 未验证用户是否真的需要AI推荐
  - 缺乏数据和技术能力
  - 存在更简单的解决方案
- 最小实验：
  1. 访谈10-15个用户，了解他们如何发现内容、当前痛点
  2. 测试简单方案：改进内容分类、添加编辑推荐
  3. 测量：内容发现成功率、用户满意度
- 成功指标：
  - 内容发现成功率提高30%
  - 用户满意度提升
- 如果简单方案失败，再考虑AI
- 下一步：本周开始用户访谈，2周内测试简单方案

## 关键洞察

通过产品思考三步法，我们：
1. 避免了"竞品驱动"的产品决策
2. 识别了真实问题（内容发现）而非表面方案（AI推荐）
3. 找到了更简单、更可行的解决方案
4. 设计了验证计划，而非盲目开发

这就是产品思考三步法的价值：**从反应式产品管理转向战略性产品思考**。
