# 产品思考示例

## 示例1：新功能评估

### 用户问题
"我们的用户一直要求添加社交分享功能。我们应该做吗？"

### 澄清阶段
**表面问题：** 是否应该添加社交分享功能？

**模糊术语：** "用户要求" - 多少用户？什么类型的用户？如何要求的？

**隐藏假设：**
- 社交分享会带来增长
- 用户要求的功能就是他们需要的
- 竞争对手有所以我们也要有

**真实目标：** 用户想要与他人分享内容，但我们需要理解为什么、分享什么、与谁分享。

### 拆解阶段
**基本事实：**
- 15个用户在过去3个月提到分享
- 当前没有分享功能
- 用户流失率在30天达到60%

**硬约束：**
- 开发团队已满负荷2个月
- 没有额外的营销预算

**软约束：**
- "社交产品应该有分享"的行业规范
- 产品管理层的偏好

**因果结构：**
- 高流失率 → 用户找不到价值 → 可能缺乏社交认可或分享成就的机制
- 但也可能：产品本身核心价值不够清晰

### 简化阶段
**假设负担：**
- 完整社交分享系统：高（假设用户会分享、假设分享带来增长、假设值得投资）
- 简单分享链接：低（测试用户是否真的想分享）

**可移除复杂度：**
- 多平台集成
- 分享预览定制
- 分享分析

### 决策阶段
**建议：** 不立即构建完整社交分享系统。

**原因：**
1. 请求量小（15个用户 / 3个月）
2. 未验证分享是否真的解决流失问题
3. 开发成本高

**最小实验：**
1. 在3-5个关键页面添加简单的"复制链接"按钮
2. 追踪点击率和分享后用户行为
3. 同时访谈5-10个用户，询问他们想分享什么、与谁分享、为什么

**成功指标：**
- 10%用户点击分享按钮
- 分享用户留存率提高20%
- 用户明确表达分享动机

**下一步：** 本周实施复制链接按钮，2周后评估数据。

## 示例2：MVP范围界定

### 用户问题
"我们正在构建一个远程团队协作工具。MVP应该包含什么功能？"

### 澄清阶段
**表面问题：** MVP功能列表是什么？

**真实问题：** 用最少功能验证什么核心价值主张？

**重构后问题：** 远程团队协作的最大痛点是什么，我们如何用最简方案验证我们能解决它？

### 拆解阶段
**真实目标：** 帮助远程团队更高效协作。

**问题组件：**
- 用户：分布式团队、跨时区
- 任务：沟通、文件共享、项目管理
- 痛点：时差、沟通异步、信息分散

**基本事实：**
- 市场已拥挤（Slack、Teams、Notion、Asana）
- 创始团队有远程协作经验
- 初始目标：小型技术团队（5-20人）

**硬约束：**
- 2个人开发团队
- 3个月开发时间
- 自力更生（无外部资金）

**软约束：**
- "应该有移动App"
- "应该有视频会议"

### 简化阶段
**当前想法的假设负担：**
- 全功能协作平台：高（假设我们可以与成熟产品竞争）
- 专注单一用例：低（假设我们在特定场景可以更好）

**可移除复杂度：**
- 移动App（Web优先）
- 视频会议（集成现有工具）
- 复杂权限系统

### 决策阶段
**MVP建议：** 专注"异步日常更新"这一单一用例。

**MVP功能：**
- 每日站队（文字版）
- 时区感知的自动提醒
- 简单进度更新（红/黄/绿状态）
- 基础历史记录

**排除的功能：**
- 实时聊天
- 文件共享
- 视频会议
- 移动App
- 复杂项目管理

**原因：**
- 远程团队的最大痛点之一是时差导致的信息不同步
- 现有工具大多是同步沟通，不适合异步
- 范围小，2人团队可以在3个月完成
- 容易验证（用户是否真的用它并付费？）

**验证计划：**
1. 招募10个远程团队试用beta版
2. 测量：每日活跃率、30天留存、付费意愿
3. 目标：40% DAU、60%留存、20%愿意付费≥$10/月

**下一步：** 创建原型，开始用户访谈验证"异步日常更新"是否真的是痛点。

## 示例3：产品复杂度简化

### 用户问题
"我们的产品变得很复杂，用户抱怨难用。我们该怎么办？"

### 澄清阶段
**表面问题：** 如何简化复杂产品？

**模糊术语：** "复杂"、"难用" - 具体指什么？

**隐藏假设：**
- 需要大重构
- 需要更好的UI/UX
- 所有功能都必要

**真实目标：** 提高用户可用性和满意度，可能通过减少而非优化。

### 拆解阶段
**基本事实：**
- 产品有47个功能
- 新用户注册率：60%完成注册
- 新用户激活率：15%完成首次关键任务
- 用户反馈："太多选项"、"不知道从哪开始"

**硬约束：**
- 有些用户依赖某些功能
- 不能立即删除所有功能

**因果结构：**
- 功能太多 → 选择过载 → 用户困惑 → 放弃
- 功能隐藏在菜单 → 用户找不到 → 认为功能不存在

**虚假复杂度：**
- 复杂度不是技术性的（代码可能还好）
- 复杂度是认知性的（用户心理负担）

### 简化阶段
**假设负担：**
- "用户需要更多功能才满意"：高假设，低证据
- "用户需要更简单的体验"：低假设，高证据（数据支持）

**可移除：**
- 使用率<5%的功能（12个功能）
- 重复功能（3组）
- "以防万一"功能（8个）

**必须保留：**
- 核心用户流程（前20%功能覆盖80%使用）

### 决策阶段
**建议：** 渐进式简化，而非大重构。

**三阶段计划：**

**阶段1（立即）：**
- 隐藏低使用率功能（移至"高级"菜单）
- 简化新手引导（只展示核心3个功能）
- 添加"快速开始"模板

**阶段2（1个月内）：**
- 正式移除使用率<2%的功能
- 合并重复功能
- 重新组织信息架构（基于用户任务）

**阶段3（3个月内）：**
- 基于用户行为重新设计核心流程
- 移除"高级"菜单（如果证明不需要）

**验证：**
- 测量激活率、支持票据、用户满意度
- 目标：激活率提升至35%，支持请求减少30%

**风险：**
- 部分用户可能抱怨功能消失
- 缓解：提前通知，提供反馈渠道，考虑保留为"可选插件"

**下一步：** 本周实施阶段1，2周后评估数据。

## 关键要点

所有示例共享的模式：

1. **从表面问题到真实问题** - 重新框架用户的初始问题
2. **暴露假设** - 明确说出未声明的假设
3. **基于事实** - 使用数据、观察、证据，而非观点
4. **简化思维** - 识别和移除不必要的复杂度
5. **可操作输出** - 以具体的下一步、实验、验证计划结束

记住：目标是更好的产品决策，而不是更聪明的分析。
