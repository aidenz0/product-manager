# 输出模式参考

## 结构化响应格式

使用这些模式确保产品思考输出清晰、可操作、一致。

## 模式A：快速重构输出

用于简短、单次交互的问题。

```markdown
## 真实问题
[陈述用户实际试图解决的问题，而非他们如何框架它]

## 隐藏假设
[列出问题框架中的关键未声明假设]

## 主要约束
[识别最重要的硬约束]

## 更简单结论
[在去除不必要复杂度后的清晰结论]

## 下一步行动
[一个立即的、具体的下一步]
```

## 模式B：结构化产品分析输出

用于新产品、功能设计、MVP界定。

```markdown
## 真实用户目标
[用户试图完成什么？为什么重要？]

## 关键假设
[需要验证的核心假设列表]

## 基本事实
[我们已知什么（证据、数据、观察）？]

## 约束分析
### 硬约束
[不可协商的限制]
### 软约束
[可挑战的惯例或偏好]

## 方案选项
### 选项A：[名称]
- 描述：[简述]
- 假设负担：[高/中/低]
- 复杂度：[高/中/低]

### 选项B：[名称]
- 描述：[简述]
- 假设负担：[高/中/低]
- 复杂度：[高/中/低]

## 产品方案/MVP建议
[推荐方案及理由]

## 验证计划
### 第一步
[具体的第一步行动]

### 成功指标
[如何知道方案有效？]

### 风险
[可能出错什么？如何缓解？]
```

## 模式C：需求审计输出

用于审查现有产品需求或功能设计。

```markdown
## 原始需求
[用户提出的需求]

## 真实问题
[实际试图解决的问题]

## 薄弱假设
[需求中未验证或可疑的假设]

## 缺失证据
[需要什么证据支持这个需求？]

## 更简单设计
[达到相同目标的更简单方式]

## 优化判断
[优化后的建议]

## 下一步
[如何验证优化建议？]
```

## 模式D：优先级决策输出

用于功能优先级或路线图决策。

```markdown
## 决策背景
[需要决定什么？约束是什么？]

## 评估标准
- 用户影响：[高/中/低]
- 业务影响：[高/中/低]
- 实现成本：[高/中/低]
- 风险：[高/中/低]

## 优先级排序
1. [功能/项目名称] - [理由]
2. [功能/项目名称] - [理由]
3. [功能/项目名称] - [理由]

## 明确排除
[不在范围内的项目及原因]

## 下一步行动
[立即执行的第一步]
```

## 模式E：假设验证输出

用于设计和验证实验。

```markdown
## 核心假设
[我们试图验证什么？]

## 预测
[如果假设为真，我们会看到什么？]

## 验证方法
- 方法：[访谈/调查/A/B测试/原型等]
- 样本：[需要多少用户？]
- 时长：[多久？]

## 成功标准
- 定量：[数字指标]
- 定性：[用户反馈信号]

## 决策规则
- 如果成功：[下一步]
- 如果失败：[替代方案]
- 如果模糊：[额外验证]

## 风险和缓解
- 风险：[什么可能出错？]
- 缓解：[如何处理？]
```

## 输出质量检查清单

在提交任何输出前，检查：

- [ ] 是否识别了真实问题？
- [ ] 是否暴露了关键假设？
- [ ] 是否区分了硬/软约束？
- [ ] 是否简化了不必要的复杂度？
- [ ] 是否提供了具体的下一步？
- [ ] 是否可操作（而非抽象分析）？
- [ ] 是否基于事实（而非 opinon）？
- [ ] 是否保留了充分性（没有天真简化）？
